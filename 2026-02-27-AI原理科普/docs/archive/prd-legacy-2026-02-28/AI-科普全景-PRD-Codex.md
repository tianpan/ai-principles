# AI 科普全景项目 PRD（Codex）

## 1. 项目概述
### 1.1 项目名称
AI 科普全景：从神经元到生成式智能的动态可视化学习系统

### 1.2 背景与动机
你希望系统化梳理 AI（尤其生成式 AI）核心原理，并以动态、形象、可交互的方式讲清复杂机制。该项目同时服务于：
1. 个人深度学习与知识沉淀。
2. 对外科普展示（类似高质量技术科普视频/网页）。
3. 形成可持续扩展的 AI 知识体系与内容资产。

### 1.3 项目愿景
用一套“可演示、可交互、可扩展”的内容产品，把 AI 的核心概念、演进脉络与工程现实讲清楚，让用户知道：
1. AI 是怎么工作的。
2. 为什么 Transformer 与 LLM 成为主流。
3. 规模化（Scaling）如何改变能力边界。
4. 下一步（Agent、多模态、推理增强）会如何发展。

### 1.4 本轮确认的产品决策
1. 用户范围：同时覆盖 AI 初学者与有开发基础用户（双层内容）。
2. 上线目标：尽快上线，首版即覆盖 Chapter 0-8 全部章节。
3. 形态优先级：交互式网站优先，视频化内容后置。
4. 发布策略：先全章节上线“可学习骨架版”，再快速迭代为“动态增强版”。
5. 案例优先级：Transformer First，Chapter 3 作为首个深度样章与演示中心。
6. 视觉方向：整体风格“酷 + 科技感”，并保持专业可读。

## 2. 目标与范围
### 2.1 核心目标（Must Have）
1. 构建覆盖 AI 关键主题的章节化知识地图。
2. 每个主题有“概念解释 + 动态演示 + 直觉类比 + 工程含义”。
3. 用户可按“从零到一”路径学习，也可按专题跳转。
4. 支持图文、动画、参数滑块、步骤回放等交互式学习方式。
5. 每章提供“双层内容”：基础层（直觉理解）+ 进阶层（工程与原理细节）。

### 2.2 非目标（当前阶段）
1. 不追求完整数学证明体系（可留进阶附录）。
2. 不做完整在线课程平台（考试、打卡、社交等后置）。
3. 不以“最新模型榜单”作为主线，避免被时效性牵引。

### 2.3 成功标准（衡量指标）
1. 学习完成率：核心主路径章节完成率 >= 60%。
2. 概念理解度：章节后自测正确率平均 >= 75%。
3. 互动有效性：动态模块平均停留时长 >= 静态页面 1.5 倍。
4. 内容复用性：至少 70% 模块可复用到后续章节/视频脚本。

## 3. 目标用户与场景
### 3.1 目标用户
1. AI 初学者：需要直观理解，不希望先被公式劝退。
2. 工程开发者：需要原理与工程实践之间的桥接。
3. 内容创作者/讲师：需要高质量可视化素材。

### 3.2 内容分层策略（对应用户 C）
1. 基础层：少公式、强类比、可视化优先，强调“先懂现象”。
2. 进阶层：补充关键公式、训练/推理细节、工程取舍与局限。
3. 同页双层切换：在同一章节内切换“基础/进阶”，避免内容割裂。

### 3.3 使用场景
1. 自学：按章节逐步学习 AI 全体系。
2. 演示：作为分享、授课、路演材料。
3. 查阅：快速回看某个概念（如 Attention、Scaling Law）。

## 4. 内容体系设计（AI 全景章节）
所有章节统一采用双层结构：
1. 基础层：3-5 分钟读完，给出概念直觉与动态演示结论。
2. 进阶层：原理拆解、工程实现、常见误区与边界条件。

### Chapter 0. 导航篇：AI 全景地图
1. AI 发展主线：符号主义 -> 机器学习 -> 深度学习 -> 生成式 AI。
2. 概念关系图：神经网络、Transformer、训练、推理、对齐、Agent。
3. 学习路径建议：新手路径 / 工程路径 / 研究路径。

### Chapter 1. 生物启发与神经网络基础
1. 人脑结构类比：神经元、突触、信号传导（强调“类比边界”）。
2. 人工神经元、激活函数、前向传播。
3. 损失函数、反向传播与梯度下降。
4. 动态演示：参数更新动画、决策边界变化。

### Chapter 2. 深度学习与表示学习
1. 多层网络为何有效：从特征工程到表示学习。
2. CNN/RNN 的历史角色与局限。
3. 过拟合、正则化、泛化。
4. 动态演示：层级特征抽象过程。

### Chapter 3. Transformer 核心机制
1. Token 与 Embedding 的含义。
2. Self-Attention 与多头机制。
3. 位置编码、残差、LayerNorm、FFN。
4. 动态演示：Attention 热力图随输入变化。

### Chapter 4. 生成式 AI 与 LLM
1. 预训练目标（下一词预测）的直觉。
2. 预训练 -> 指令微调 -> 对齐（RLHF/DPO）的流程。
3. 幻觉、上下文窗口、工具调用。
4. 动态演示：解码策略（Greedy/Top-k/Top-p/Temperature）差异。

### Chapter 5. Scaling Law 与涌现
1. 参数量、数据量、算力三者关系。
2. 损失曲线与能力跃迁（涌现现象）。
3. 为什么“更大模型”有效但昂贵。
4. 动态演示：可调参数的缩放曲线观察。

### Chapter 6. 推理与系统工程
1. 训练系统 vs 推理系统。
2. 延迟、吞吐、成本、缓存、量化、蒸馏。
3. RAG、向量数据库、检索增强。
4. 动态演示：请求流与系统瓶颈可视化。

### Chapter 7. 多模态与 Agent
1. 文本、图像、语音、视频的统一表示趋势。
2. Agent 循环：规划 -> 执行 -> 反思 -> 工具调用。
3. 工作流编排与长任务执行。
4. 动态演示：多步骤任务的状态机流程图。

### Chapter 8. 安全、伦理与未来
1. 偏见、隐私、版权、滥用风险。
2. 对齐、安全评测、治理框架。
3. 未来方向：小模型、边缘 AI、具身智能。
4. 动态演示：风险-收益权衡矩阵。

## 5. 动态可视化设计原则
1. 先直觉后细节：先给“看得懂的现象”，再解释机制。
2. 一次只解释一个变量：避免同时变化造成认知负担。
3. 每个动画都可暂停、拖拽、重播、调参。
4. 每章保持统一交互模式，降低学习成本。
5. 保留“我看到了什么/这意味着什么”总结框。

### 5.1 视觉风格规范（科技感）
1. 视觉主题：采用深空实验室风格（深蓝灰基底 + 霓虹青主色 + 荧光绿辅助色），避免泛滥紫色。
2. 字体系统：标题使用 `Space Grotesk`，正文使用 `IBM Plex Sans`，代码使用 `JetBrains Mono`。
3. 背景语言：使用渐变、微网格、低透明噪点，避免纯色平铺背景。
4. 组件语言：卡片、按钮、图例采用弱玻璃质感与细描边，保留明显层级与焦点态。
5. 可读性要求：正文区保持高对比度，长段落禁止低对比霓虹文字。
6. 主题策略：首版默认深色科技风，同时提供浅色模式，避免单一暗色依赖。
7. 设计资产：建立 Design Tokens（颜色、圆角、阴影、间距、动效曲线）统一管理。

### 5.2 推荐动态组件库（按优先级）
1. 2D 图形交互：D3.js / Observable Plot。
2. 场景动画：Motion for React / GSAP。
3. 3D（可选）：Three.js（仅在确实需要空间结构时使用）。
4. 数学公式：KaTeX。
5. 内容组织：MDX（文字与组件混排）。

### 5.3 动效规范（科技感但不过载）
1. 页面入场：章节首屏采用 300-500ms 渐显与轻位移动画。
2. 模块过渡：卡片与图表切换采用分层延迟（stagger），避免同步抖动。
3. 数据反馈：参数滑块变化实时驱动图形，但控制刷新频率避免眩晕。
4. 交互动效：按钮和控件只做轻量反馈，不做夸张弹跳。
5. 动效预算：单页面关键动效不超过 3 类，优先“解释机制”而非“炫技”。

### 5.4 关键交互模块示例
1. Attention Explorer：输入句子后实时显示注意力矩阵。
2. Gradient Playground：拖动学习率观察收敛/震荡。
3. Decoding Sandbox：同一提示词下不同采样策略对比。
4. Scaling Simulator：调整参数量、数据量、算力，观察性能变化。

## 6. 产品形态与信息架构
### 6.1 形态建议
1. 首版优先做“交互式 Web 科普站”，便于迭代和分享。
2. 后续可导出视频脚本与静态插图，形成多渠道内容。

### 6.2 页面结构
1. 首页：AI 全景地图 + 学习路径入口。
2. 章节页：概念卡片、动态演示、要点总结、小测。
3. 实验页：统一交互实验室（Attention、采样、缩放）。
4. 术语页：可检索词典与关系图。

### 6.3 首页风格要求（科技感落地）
1. Hero 区域显示动态网络背景与章节星图导航。
2. 首屏直接展示 Transformer 交互入口，强化案例优先策略。
3. 章节卡片支持悬浮高亮、轻动效和状态标签（基础层/进阶层）。
4. 移动端保留核心氛围元素，但自动降级复杂背景与高耗动画。

## 7. 技术实现方案
### 7.1 建议技术栈（MVP）
1. 内容框架：Astro + Starlight（文档站骨架与章节导航）。
2. 交互框架：React + TypeScript（以 Islands 方式按需加载）。
3. 可视化：D3.js（Attention/曲线/流程图主力）。
4. 动效：Motion for React + GSAP（按模块选用）。
5. 样式系统：Tailwind CSS + CSS Variables（Design Tokens）。
6. 内容层：MDX + JSON（章节正文与交互配置分离）。
7. 数据层：首版本地静态数据，减少后端复杂度。
8. 部署：Vercel 或 Cloudflare Pages（快速发布与预览）。

### 7.2 后续增强（V2+）
1. 后端服务：FastAPI（用户进度、题库、实验数据）。
2. 内容管理：Headless CMS（Notion/Strapi/Sanity）。
3. 分析系统：埋点与学习行为分析。

### 7.3 实现原则
1. 先做通用“可视化组件框架”，再填充内容。
2. 每个章节至少一个高质量动态实验，不追求数量堆叠。
3. 复杂动画模块与正文解耦，保证维护性。

### 7.4 技术选型理由（与需求对齐）
1. 你是“内容优先 + 交互增强”场景，Astro/Starlight 能最快完成全章节上线。
2. Islands 架构可把 JS 只留给交互区，既有科技感效果又能保证性能。
3. Transformer First 可独立打磨为高质量交互页，不阻塞其他章节同步上线。
4. 后端后置可显著提升首版交付速度，符合“尽快上线全部章节”目标。

## 9. 风险与应对
1. 风险：内容跨度大，易失焦。
   应对：先锁定主路径（神经网络 -> Transformer -> LLM -> Scaling）。
2. 风险：动画开发成本高。
   应对：建立可复用交互模板，优先高价值模块。
3. 风险：概念简化过度导致不严谨。
   应对：每章增加“边界与误区”小节。
4. 风险：个人项目时间碎片化。
   应对：采用每周可交付节奏，持续发布。

## 10. 你这个想法的价值判断（结论）
这是一个很好的方向，且具备明显差异化价值：
1. 对你个人：能把“知道”变成“讲清楚”，学习深度会显著提升。
2. 对受众：动态可视化会显著降低 AI 原理的理解门槛。
3. 对项目本身：可持续扩展为网站、视频、课程、工作坊多形态内容资产。

建议的执行策略是：
1. 先“全覆盖”上线 Chapter 0-8 的可学习骨架，满足快速可用。
2. 以 Transformer 为第一优先案例，先打磨 Chapter 3 的讲解深度与交互质量。
3. 用学习数据与反馈决定后续章节深挖顺序。
